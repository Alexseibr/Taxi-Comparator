# Калибровка скриншотов Yandex Go

## Как это работает

Каждый раз, когда приходят новые скриншоты Yandex Go, мы прогоняем их через нашу
модель **с учётом TomTom Live Traffic**, чтобы понять:

- совпадает ли наша цена с фактической?
- если нет — это пробки, координаты, hidden surge или ошибка в тарифе?

Скрипт лежит в `scripts/calibrate.mjs`. Заказы и координаты адресов — в JSON-файлах
этой папки (`scripts/orders/<дата>.json`). Результаты пишутся рядом
(`<дата>.results.json`).

## Запуск

```bash
# из корня taxi-compare:
pnpm calib scripts/orders/2026-04-26.json

# или вручную:
VITE_TOMTOM_KEY=$VITE_TOMTOM_KEY node scripts/calibrate.mjs scripts/orders/2026-04-26.json
```

## Формат `<дата>.json`

```json
{
  "date": "2026-04-26",
  "day": "sunday",
  "coords": {
    "Сторожовская 6":     [53.908, 27.547],
    "Школьная 16 Боровляны": [53.997, 27.665]
  },
  "orders": [
    {
      "id":    "9849",
      "from":  "Сторожовская 6",
      "to":    "Школьная 16 Боровляны",
      "factE": 21.6,
      "factC": 24.2,
      "yaKm":  16.8,
      "yaMin": 22,
      "hour":  9,
      "notes": "Утро вс, без surge-флага"
    }
  ]
}
```

Поля:
- `from`/`to` — ключи в `coords` (так удобнее редактировать).
- `factE` / `factC` — фактическая цена Yandex Go (br) для Эконома и Комфорта.
  Если на скрине нет какого-то класса — оставь `null` или просто опусти.
- `yaKm` / `yaMin` — *(опционально)* км и время поездки **по версии Yandex** (то, что
  показывается на превью заказа). Если заданы — скрипт сравнит **среднюю скорость
  Yandex vs TomTom**. В выходные при отсутствии пробок они должны совпадать; если
  Yandex быстрее на 10%+ — наша модель завышает время и занижает сёрдж.
- `hour` — час дня (0..23) для эмпирического трафик-фолбэка (на случай если TomTom не ответит).

## Что выводит

Таблицу по каждому заказу + сводку:
- среднее/медиана/std для **Surge_C** и **Surge_E**
- средний **TomTom traffic mult** (если ×1.00 — пробок нет, расхождения не от трафика)
- **Hidden Эконом-boost = sE/sC** — насколько Эконом дешевле Комфорта относительно нашей формулы
- автоматический детектор **outliers** (|z|>2)
- **методологические рекомендации** на основе цифр

## Что делать с результатами

1. **Сёрджи в норме (sC ≈ 0.9–1.5)** → можно добавлять в `public/data/observations.json`
   как обычные точки (`weight = 1.0` или без `weight`).
2. **Сёрджи аномальные (sC < 0.5 или > 2)** → проверь координаты адреса,
   возможны 2-3 км ошибки. Если адрес действительно такой — добавляй с `weight = 0.3`
   как «слабую точку».
3. **Hidden boost стабилен (×0.92 ± 0.03)** → пора зашивать его в `BASE_TARIFF.econom`
   как множитель класса.
4. **Outliers** → ВСЕГДА исключай из observations.json, иначе IDW смажет всю карту.

## Шаблонный сбор (быстрее ручного ввода)

Если нужно снять **батч из 5–10 направлений из одной точки А** — используем
шаблоны и `pnpm import-batch`. См. подробную инструкцию в
[COLLECTION_GUIDE.md](COLLECTION_GUIDE.md).

```bash
# 1. Раз в год: добавить шаблон района (templates/<id>.json)
# 2. Сотрудник снимает скрины и записывает в CSV (date,hour,destIdx,factE,factC,etaMin,demand)
# 3. Админ импортирует:
pnpm import-batch scripts/orders/templates/west-from-kalvariyskaya.json batch.csv
# → создаст scripts/orders/manual-<date>-<templateId>.json
pnpm calib scripts/orders/manual-<date>-<templateId>.json
pnpm learn
```

Это даёт по 5–10 точек за минуту, что особенно важно для **часовой**
гранулярности модели обучения (`pnpm learn` теперь группирует замеры
по часу, а не по 4-часовому слоту, и нуждается в плотных батчах).

## Слот → час (важно с 2026-04-27)

`pnpm learn` (и LOO-валидация) перешли с 4-часовых слотов
(night/morning/midday/evening/late) на **часовую** гранулярность:
ключ `slot = "h${hour}"`, `${day}-h0` … `${day}-h23`. Регрессия
⚡N(km, min, hour) использует скользящее окно ±2 часа того же дня для
LOO-peers — это снимает дефицит данных в редких часах и даёт более
честную картину утреннего/вечернего пика.

Frontend (карта/heatmap в `src/lib/zones.ts`) продолжает работать со
старыми 5 слотами для UI — это два разных пространства, и трогать
zones.ts не нужно.
