# Методология формирования тарифов RWB Taxi (v3)

*Документ описывает всё, что система выявила, замеряя цены Yandex Go и восстанавливая их формулу обратной инженерией. Обновляется по мере накопления данных через `pnpm calib && pnpm learn`.*

**Текущая версия:** v3 (плоская baza, бинарная hb). Дата калибровки: 26.04.2026 (10-й прогон 15:42–15:49 — пакет из 32 заказов из 3 опорных точек: Немига 3 / Жуковского 16 / Кальварийская 33; плюс отдельная подсборка из аккаунта B 15:58 на 2 заказа — для аналитики кросс-аккаунтной персонализации Yandex, не влияющая на обучение).
**Dataset:** 110 заказов из 10 прогонов калибровки участвуют в обучении; ещё 2 заказа аккаунта B (карта •9302) вынесены отдельно для аналитики кросс-аккаунтной персонализации. Из 110 обучающих 60 имеют открытый ⚡N (дальность поездок 1..150 км). MAE модели hb на 109 точках с валидным sE/sC ≈ 0.033, LOO MAPE ≈ 33.7% (в пределах ±10% — 18/60, ±20% — 26/60).

---

## Базовая формула цены

```
factC = minimum_C × surge_C(day, slot, cell, km, min, traffic)
factE = minimum_E × surge_E ≈ factC × hidden_boost
```

Yandex показывает пользователю одну величину `factC`, но методология восстановила её структуру из 4-х независимых множителей.

---

## Слой 1 — Базовый тариф (плоская монета)

```
minimum_C = 10 br   (Комфорт, всегда, на любом расстоянии)
minimum_E =  9 br   (Эконом = 10 × hidden_boost ≈ 0.89)
pickup    = 0
perKm     = 0
perMin    = 0
```

### Открытие v3

У Yandex `pickup`, `perKm` и `perMin` **фактически нулевые**. Baza плоская = **9.63 br ± 0.21** на ВСЕХ дистанциях, включая аэропорт MSQ (44 км → baza_Y = 9.93 br). Длина и время маршрута зашиты не в тариф, а в `surge` — поэтому ⚡N в UI коррелирует с км (отношение ⚡N/км ≈ 0.13–0.20). Иконка «×N» в Yandex — это **полный множитель цены к плоской монете 10 br**, а не «множитель спроса».

### Sanity-check регрессии (доказательство плоской baza)

Регрессия `baza_Y = pickup + perKm·km + perMin·min` по 60 заказам с открытым ⚡N:

| Параметр | Восстановлено | Вердикт |
|---|---|---|
| `pickup` | 9.63 br | ≈ minimum_C ✅ |
| `perKm`  | **−0.0038** br/км | < 0.05 → ноль ✅ |
| `perMin` | **0.0123** br/мин | < 0.05 → ноль ✅ |
| MAE      | 0.124 br | модель почти идеально объясняет цену |

**Вывод:** гипотеза v3 (плоская baza) подтверждена. Снижать `perKm`/`perMin` бесполезно — они уже ноль.

## Слой 2 — Surge-карта по времени и месту

`surge` — мультипликатор цены, зависит от:

| Измерение | Значения | Источник |
|---|---|---|
| `day`  | weekday / saturday / sunday | Календарь |
| `slot` | night (0–6) / morning (7–10) / midday (11–14) / evening (15–19) / late (20–23) | Час заказа |
| `cell` | Hex H3 ячейка координат отправления | Геокод |
| `km`, `min` | Длина и продолжительность маршрута | OSRM |

### Эмпирически зафиксированная динамика (только воскресенье)

| Время | n | mean sC (наша модель) | Открытый ⚡N со скрина | Комментарий |
|---|---|---|---|---|
| 10:33 (morning) | 18 | **0.59** | mean 0.96 (n=8) | Спокойное утро |
| 11:12–11:24 (midday «пик») | 8 | **3.46** | mean 3.46 (n=8) | Открытые ⚡N на Cmf, спрос ×3.6 от утра |
| 12:08–12:10 (midday «спад») | 9 | **1.83** | — (нет ⚡N на Cmf) | Спрос упал ВДВОЕ за 45 минут |

**Ключевое открытие 5-го прогона (12:08):** в midday зафиксирован не только пик, но и **фаза спада**. К 12:08 «Высокий спрос» лейбл ещё висит, но открытые ⚡N на Комфорте уже исчезли — Yandex показывает только ⚡ на Экономе. Это значит, что midday-слот — двухфазный: пик 11:00–11:30 → спад 12:00+.

Такая внутрислотовая динамика **не моделируется** одним числом: `surge-map.json` усредняет фазы и в midday показывает std=0.79 при mean=1.32.

### Регрессионная модель ⚡N(km, min, slot)

Внутри каждого `(day×slot)` фитим линейную модель:

```
sunday-morning: ⚡N ≈ 0.60 + 0.321·km − 0.154·min   (MAE=0.264)
sunday-midday : ⚡N ≈ 0.84 + 0.073·km + 0.049·min   (MAE=1.103)
```

Modelling-замечание: для midday MAE (1.1) сравним со std (2.0) — это значит, что главный сигнал лежит в `cell` (район отправления), а не в `km/min`. Регрессия здесь — не предсказатель, а первая аппроксимация. По-настоящему интерполяция работает через **IDW (inverse distance weighting)** по H3-ячейкам, ограниченную МКАД.

### Что предстоит замерить

Все слоты кроме воскресных утра/обеда — **пустые**. Приоритет следующих замеров:

1. `weekday-morning` (07–10) — час пик в будни
2. `weekday-evening` (15–19) — обратный час пик
3. `friday-evening` — пятничный пик
4. `saturday-night` — выходные ночью

## Слой 3 — Hidden Эконом-boost

Yandex применяет к Эконому **скрытую скидку** относительно Cmf, не показывая её в UI:

```
hidden_boost = (factE / minimum_E) ÷ (factC / minimum_C) = sE / sC
```

### Зависимость hb от sC: бинарная функция (опровержение «горба»)

Анализ **109 замеров** (10 прогонов вс 26.04.2026, MAE ≈ 0.033) подтверждает, что зависимость `hb(sC)` бинарная, а не «горб». Прежняя гипотеза «горба с пиком hb≈0.99 в зоне 1.5–2.5» была артефактом малой выборки и больше не воспроизводится:

| sC (наш v3) | n | mean hb | Что это значит |
|---|---|---|---|
| < 1.0   | 7  | ×0.891 | классическая скидка ~−11% (стимул конверсии) |
| 1.0–1.3 | 10 | ×0.969 | сжатая скидка ~−3% |
| 1.3–1.6 | 8  | ×0.948 | то же |
| 1.6–2.0 | 13 | ×0.965 | то же (бывший «пик» теперь стабильно в плато) |
| 2.0–2.5 | 13 | ×0.963 | то же |
| 2.5–4.0 | 8  | ×0.960 | то же |
| 4.0–7.0 | 5  | ×0.950 | междугородние; пограничный 9947 (Минск→Фаниполь) дал hb=0.892, тянет среднее вниз |
| > 7.0   | 1  | ×0.979 | междугородние, чуть выше |

### Реальная модель: ДВА режима скидки

У Yandex срабатывают только два режима, никакого «исчезновения и возврата скидки» нет:

- **(а) низкий спрос (sC < 1.0):** hb = 0.89 — классическая «−11%» скидка как стимул, чтобы клиент вообще что-то заказал
- **(б) есть спрос (sC ≥ 1.0):** hb ≈ 0.96 — сжатая «−4%»; Yandex не отдаёт большую скидку, раз клиент уже готов платить

Между ними короткий переход 1.0–1.2 для гладкости (без скачка). Для междугородних (sC > 5) скидка чуть мягче — hb ≈ 0.97.

### Как boost применяется в v3

```ts
function hiddenBoost(sC: number): number {
  if (sC < 1.0) return 0.89;                       // низкий спрос → стимул
  if (sC < 1.2) return 0.89 + (sC - 1.0) * 0.35;   // плавный переход 0.89 → 0.96
  if (sC < 5.0) return 0.96;                       // основной плато
  return 0.97;                                     // междугородние
}
```

`predictEconom(sC)` всегда возвращает `sE = sC × hiddenBoost(sC)` и кладёт результат в поле `hidden`, которое перекрывает observation-based econom при отрисовке цены. Это сделано сознательно: hb зависит от sC через бинарный режим, и наблюдение в одной фазе не предсказывает поведение в другой.

### Урок методологии

Гипотеза «горба» родилась на 35 замерах, где бакет 1.5–2.5 содержал n=15 точек со счастливо высоким hb=0.989. После добавления 16 новых замеров из Брилевичей (вс 13:30, фаза «спад») n стало 25, а hb упал до 0.967 — пик исчез. **Никогда не доверять пиковой точке с n<20**, особенно если соседние бакеты гораздо ниже.

Прежние бамп-коэффициенты (×1.05, ×1.10, ×1.20 для surge ≥ 1.2/1.5/1.8) убраны ещё на 5-м прогоне — они выводили hb выше 1.0, чего не наблюдалось ни в одном замере.

## Слой 4 — Поправка на трафик (TomTom Live)

`ttMult` — мультипликатор замедления от пробок, измеряется по TomTom Live в N точках вдоль маршрута OSRM.

**Текущий статус:** ttMean ≈ 1.00..1.12 на 110 замерах (все — воскресенье без значимых пробок; в 10-м прогоне evening 15:42-15:49 ttMult колебался 1.00..1.03). Корреляцию `surge ↔ ttMult` пока не построить. Слой работает «вхолостую» — будет активен после первых замеров в час пик.

### Двойное использование TomTom

1. **Real-time на карте** — окрашивает участки маршрута красным/жёлтым/зелёным.
2. **Калибровочный сигнал** — если ⚡N растёт одновременно с `ttMult`, формула учится связывать их линейно.

---

## Что мы НЕ воспроизводим (явно)

- **ML-предсказание спроса** Yandex (плотность заказов / водителей) — у нас вместо этого статичная surge-карта по 15 слотам, обновляемая замерами.
- **Зональные коэффициенты Yandex** (центр / окраина / пригород) — у нас аналог через H3-ячейки и `zoneMultipliers` в `zones.ts`.
- **Аукцион между водителями** — не моделируем, считаем что цена = тариф × сёрдж.
- **Лояльность пользователя, промокоды, «Plus»** — игнорируем.

## Цикл накопительного обучения

Это **самое главное в методологии** — она не статичная, каждый новый скрин её улучшает:

```
1. Скрин Yandex Go            → orders/<date>.json    (вручную)
2. pnpm calib                 → OSRM + TomTom + восстановление baza_Y
3. pnpm learn                 → 4 слоя обучения, append в changelog
4. pnpm apply                 → отчёт о состоянии методологии
```

### Что делает каждый прогон `learn`

- **L1 SANITY** — перепроверяет регрессию `baza_Y ~ km, min`. Если `perKm/perMin > 0.05` — гипотеза v3 опровергнута, нужно менять тариф.
- **L1 SURGE MODEL** — фитит ⚡N(km, min, slot) внутри каждого слота → `scripts/learned/surge-model.json`.
- **L2** — обновляет surge-mean для каждой клетки `day × slot × cell`.
- **L3** — пересчитывает hidden boost по слотам.
- **L4** — корреляция surge ↔ ttMult (когда появятся данные с пробками).
- Append-запись в `scripts/learned/changelog.md` (никогда не перезаписывается).

### Когда `apply` вернёт `❌`

`pnpm apply` отчитывается о состоянии методологии. Если sanity-check вдруг покажет `perKm > 0.05`, команда выйдет с ошибкой и попросит вручную пересмотреть `BASE_TARIFF` в `src/lib/zones.ts`.

## Коррекция модели после 10-го прогона (вс 26.04.2026, 21:30)

**Проблема:** LOO MAPE = 33.7%, в ±10% попадало только 18/60 заказов. Главные промахи:
- Дальние загородные (>50 км): pred ⚡6.0 (упор в hard-cap), факт ⚡10.9–31.6 → недооценка -45..-81%.
- Городские в бакете ⚡1-2: регрессия в каждом слоте видела дальние outliers и тянула наклон вверх → переоценка +30..+50% на коротких.

**Что поменяли в `learn.mjs` (LOO):**
1. **Сегментация peers по дальности.** Для target.km > 30 регрессия учится только на дальних peers (km > 20). Для коротких/средних — только на peers km ≤ 30. Раньше всё было в одной корзине.
2. **Расширили cap для дальних** с [0.5, 6.0] до [0.5, 35.0] — наблюдали ⚡31.6 на Морочь.
3. **Добавили fallback `mean-by-long-segment` / `mean-by-short-segment`** — если в сегменте < 3 peers, берём среднее по сегменту.

**Результат:**
| Метрика | До | После | Δ |
|---|---|---|---|
| Overall MAPE | 33.7% | **28.0%** | -5.7 пп |
| В ±10% | 18/60 | **25/60** | +7 |
| В ±20% | 26/60 | **34/60** | +8 |
| ⚡<1 (n=5) | 50.5% | 47.3% | ≈ |
| ⚡1-2 (n=33) | 24.5% | **20.2%** | -4.3 пп |
| ⚡2-3 (n=12) | 39.9% | **28.1%** | -11.8 пп |
| ⚡≥3 (n=10) | 48.1% | **44.2%** | -3.9 пп |

**Конкретные исправления:**
- 0004 Жуковского→Момо: -43% → **-7.7%** (попал в ±10%)
- 9991 Сутоки 80 км: -45% → -66% (хуже — мало дальних peers, регрессия неустойчива)
- 9982 Вилейка 98 км: -68% → +36% (модель больше не упирается в cap, но переоценила)
- 9987 Морочь 103 км: -81% → -34% (всё ещё outlier, факт ⚡31.6 — крайне редкое значение)

**Известные новые промахи** (раньше скрывались под cap=6.0):
- 9880 Ермака → Жилуновича: pred ⚡4.94 vs факт ⚡2.17 (+128%) — sleeper-эффект, нужно больше точек
- 9873 → Уручье: pred ⚡3.14 vs факт ⚡1.48 (+112%) — то же
- Эти переоценки видны теперь честно; раньше они тоже были, но прятались за hard-clamp.

**Frontend (zones.ts):** трогать не требуется. Глобальный `SURGE_BOUNDS` уже {min: 0.3, max: 10.0} — для real-time предсказаний MapDashboard этого хватает (карта не показывает Морочь/Сутоки). Калибровочные коэффициенты `baza_Y=9.91`, `hb` по бакетам — без изменений.
