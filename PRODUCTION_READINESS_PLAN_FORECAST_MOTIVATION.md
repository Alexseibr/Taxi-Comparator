# План перехода к production: прогноз + мотивации водителям

Дата: 2026-05-12
Статус: draft v1

## Цель
Запустить production-контур, в котором прогноз спроса управляет мотивациями водителей, а эффект измеряется по KPI и unit-экономике.

## KPI успеха
- ETA p50: снижение минимум на 8% в приоритетных слотах.
- Acceptance Rate: рост минимум на 5%.
- Доля «пустых» часов supply gap: снижение минимум на 15%.
- Ограничение бюджета мотиваций: не выше заданного weekly cap.
- Прогноз: MAE по surge-мультипликатору в целевом диапазоне, без деградации >10% WoW.

## Фаза 1 (1–2 недели): Технический фундамент
1. Подключить persistence для `city-control-tower` роутов.
2. Добавить таблицы версий моделей/правил мотиваций.
3. Ввести обязательные audit-поля (created_by, updated_by, reason).
4. Настроить базовые метрики и алерты (ошибки API, latency, свежесть прогноза).

**Артефакты фазы:**
- Миграции БД.
- Стабильные CRUD-endpoints.
- Дашборд “Forecast health”.

## Фаза 2 (1–2 недели): Контур прогноза
1. Регламентный job пересчёта прогноза (с retry/backoff).
2. Backtesting: ежедневный отчёт MAPE/MAE по часам/дням.
3. Алерты деградации точности и fallback при недоступности weather API.
4. Версионированный rollout surge-модели (A/B).

**Артефакты фазы:**
- Планировщик пересчётов.
- Отчёт качества модели.
- Процедура rollback модели.

## Фаза 3 (2–3 недели): Мотивации водителям
1. Правило-движок: сегменты, бонусы, caps/floors.
2. Weekly/day planning по supply-demand gap.
3. Автогенерация кампаний с ручным approve для high-budget сегментов.
4. Закрытый цикл: рекомендация → выплата → измерение эффекта.

**Артефакты фазы:**
- Конфигурируемые правила мотиваций.
- Экран планировщика кампаний.
- Отчёт эффективности кампаний.

## Фаза 4 (1–2 недели): Безопасность и операционная готовность
1. Антифрод-мониторинг по бонусам и аномалиям.
2. Финансовые лимиты (daily/weekly stop-loss).
3. RBAC + журнал критичных действий.
4. Incident runbook + on-call алерты.

**Артефакты фазы:**
- Политики лимитов и согласований.
- Runbook инцидентов.
- Чеклист Go-Live.

## Минимальный Go-Live checklist
- [ ] Все ключевые API покрыты контрактными тестами.
- [ ] Прогноз пересчитывается по расписанию, данные свежие.
- [ ] Есть дашборды: качество прогноза, KPI мотиваций, бюджет.
- [ ] Включены лимиты и антифрод-правила.
- [ ] Назначены владельцы on-call и SLA/SLO.

## Риски и смягчение
- Риск: дрейф модели из-за сезонности.
  - Смягчение: weekly backtesting + алерт на деградацию.
- Риск: перерасход бонусного бюджета.
  - Смягчение: caps + stop-loss + approve workflow.
- Риск: неполные данные.
  - Смягчение: DQ-валидации и контроль freshness.
