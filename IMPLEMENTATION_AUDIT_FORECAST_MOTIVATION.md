# Аудит: «прогноз» и «мотивации водителям»

Дата проверки: 2026-05-12

## Вывод

1. **Прогноз спроса/цен — реализован.**
   - Есть backend-логика прогноза на основе модели surge, погоды и событий.
   - Есть команды в Telegram (`/forecast`, `/прогноз`) и UI-панель прогноза на 24 часа.

2. **Мотивации водителям — реализованы частично.**
   - Есть расчёт минимально экономически оправданной цены для водителя и требуемого бонуса.
   - Есть генерация ценового коридора с компонентом `driverProtectionPrice`.
   - Но блок заявлен как MVP с ограничениями: без полноценной persistence и без weekly/promo/LLM-слоёв.

## Доказательства

### 1) Прогноз
- Telegram-команда строит 12-часовой прогноз с оценкой цены Эконом, метками дождя/снега/событий и рекомендациями.
- UI-компонент `DemandForecastPanel` показывает 24ч прогноз, уровни спроса, драйверы всплеска, погодный вклад и события.
- Базовая модель `computeSurge/forecastSurge` учитывает пики (утро/вечер), nightlife, погоду, события, волатильность маршрута и jitter.

### 2) Мотивации водителям
- API `POST /api/city-control-tower/driver-profitability/calculate` возвращает:
  - `minDriverEconomicsPrice`
  - `requiredBonus`
- API `POST /api/city-control-tower/pricing/corridor` возвращает ценовой коридор, включая `driverProtectionPrice`.
- Документация `city-control-tower` прямо помечает модуль как MVP и перечисляет ограничения.

## Что ещё не закрыто (по формулировке «задуманное»)

Если под «задуманным» подразумевался production-ready контур мотивации водителей (планирование, постоянное хранение, автоматические кампании, UI-потоки, мониторинг эффекта), то это **не полностью реализовано** в текущем состоянии:
- CRUD/persistence в роутерах пока не подключены.
- Weekly plan, promo-supply, LLM-layer и UI для этих частей — отложены на следующие итерации.

## Итоговый статус

- **Прогноз:** ✅ Да, реализован и доступен в Telegram + UI.
- **Мотивации водителям:** ⚠️ Частично (основные экономические расчёты есть, но end-to-end продуктовый контур ещё не завершён).

## Что нужно для production-версии

### 1) Данные и хранение
- Подключить реальные CRUD-операции к БД для `city-control-tower` (зоны, маршруты, наблюдения, baseline/signals, результаты расчётов).
- Ввести версионирование моделей/коэффициентов и хранение истории пересчётов.
- Добавить контроль качества данных (валидация, дедупликация, контроль пропусков, SLA загрузки).

### 2) Forecast-контур
- Перевести прогноз в наблюдаемую службу: расписание пересчётов, retry/backoff, деградация при недоступности погодного API.
- Сделать полноценную оценку качества (MAPE/MAE по слотам, backtesting по периодам, алерты по дрейфу).
- Добавить A/B-режим для сравнения версий surge-модели и безопасный rollout.

### 3) Мотивации водителям (продуктовый контур)
- Реализовать правило-движок мотиваций: сегменты водителей, типы бонусов, лимиты бюджета, caps/floors.
- Добавить weekly/day plan по supply-demand gap и автоматическую генерацию кампаний.
- Реализовать закрытый цикл: «рекомендация → запуск кампании → факт выплат → эффект на supply/ETA/acceptance». 

### 4) API/UI и операционный контур
- Довести API до production-контрактов: идемпотентность, пагинация, фильтры, audit log, RBAC.
- Добавить интерфейсы оператора: планировщик мотиваций, мониторинг KPI, журнал решений.
- Включить observability: метрики, логи, трассировка, дашборды и on-call алерты.

### 5) Безопасность и финансы
- Контроль антифрода по бонусам (аномалии, повторные выплаты, риск-профили).
- Финансовые ограждения: дневные/недельные бюджеты, stop-loss правила, approvals для крупных кампаний.
- Юридические и бухгалтерские требования: прозрачные правила начисления и отчётность по выплатам.

### 6) Критерии готовности (Definition of Done)
- Прогноз: стабильная точность на целевом окне + алерты по деградации.
- Мотивации: подтверждённый uplift по supply/ETA/CR при соблюдении unit-экономики.
- Надёжность: SLO по API/пересчётам и регламент инцидентов.
